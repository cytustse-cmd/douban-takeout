## Findings

### 1. Cookie handling is unsafe in CLI usage
- Severity: high
- Locations:
  - `README.md:57-58`
  - `README.md:250-250`
  - `douban_export.py:779-820`
  - `export_statuses_web.py:558-587`
- Problem:
  - The documented workflow encourages `--cookie 'dbcl2=...;ck=...'` on the command line.
  - Command-line arguments are commonly persisted in shell history and can be visible to other local users via process inspection.
  - `douban_export.py` falls back to plain `input()`, which echoes the full cookie in the terminal.
- Impact:
  - `dbcl2` and `ck` are live authentication credentials. Exposing them weakens account security and makes the current UX risky on shared machines, recorded terminals, or remote shells.
- How to fix:
  - Add `--cookie-file /path/to/cookie.txt` and read credentials from a file with restrictive permissions.
  - Add `--cookie-stdin` so users can pipe credentials without storing them in shell history.
  - Replace interactive `input()` with `getpass.getpass()` in `douban_export.py`.
  - Keep `--cookie` only for backward compatibility, but mark it as insecure in help text and README.

### 2. Year inference can generate wrong dates around New Year
- Severity: medium
- Locations:
  - `export_statuses_web.py:285-296`
  - `export_statuses_web.py:523-523`
- Problem:
  - When the page only shows `3月19日`-style dates, the scraper injects `datetime.now().year` directly.
  - If the export is run in January, old December posts can be normalized into future dates.
- Impact:
  - Exported timestamps become semantically wrong.
  - `my_statuses.md` and `all_statuses.md` can be sorted incorrectly.
  - Downstream consumers of the JSON/Markdown get corrupted chronology.
- How to fix:
  - Normalize into a real `date` object first, then serialize to `YYYY-MM-DD`.
  - If the parsed month/day would land in the future relative to the export date, subtract one year.
  - Store both `raw_time` and normalized `create_time`, so parsing decisions remain auditable.

### 3. End-of-pagination depends on parser output count, which can silently truncate exports
- Severity: medium
- Locations:
  - `export_statuses_web.py:380-406`
- Problem:
  - The scraper stops when `len(page_statuses) < PAGE_SIZE`.
  - That assumes the parser extracted every item on the page.
  - If Douban changes markup slightly or one item is missed by regex parsing, the scraper can falsely treat a non-final page as the last page.
- Impact:
  - Silent data loss in the exported timeline.
  - Users get incomplete backups without an explicit error.
- How to fix:
  - Detect pagination from the raw HTML instead of parsed item count, for example by checking for the next-page link or pager state.
  - Keep the current `consecutive_empty` safeguard, but do not use parsed count as the only end condition.
  - Add a consistency check: compare the number of `status-item` markers in raw HTML with parsed records, and warn if they diverge.

### 4. README and changelog overstate the cookie-unification fix
- Severity: medium
- Locations:
  - `README.md:130-137`
  - `README.md:154-160`
  - `README.md:327-331`
  - `README.md:359-363`
  - `douban_export.py:90-106`
- Problem:
  - README says both scripts now auto-detect `Safari -> Chrome`.
  - `export_statuses_web.py` does that.
  - `douban_export.py` still only tries Chrome.
- Impact:
  - Current documentation is materially inaccurate.
  - A user following the documented workflow can succeed in one script and fail in the other with the same browser state.
- How to fix:
  - Reuse the same browser-detection loop from `export_statuses_web.py` in `douban_export.py`.
  - Align function names, docstrings, and log messages across both scripts.
  - Update the changelog only after the behavior is actually unified.

## Improvement Suggestions

### UX
- Add `requirements.txt` or `pyproject.toml`.
  - Today the repo documents `pip install requests browser-cookie3`, but there is no installable dependency manifest.
  - A minimal `requirements.txt` improves first-run onboarding and CI setup.
- Add `--output` to `douban_export.py`.
  - `export_statuses_web.py` supports custom output paths, but `douban_export.py` is hard-coded to `output/`.
  - For the recommended two-step workflow, this asymmetry is unnecessary friction.
- Add a `--non-interactive` mode.
  - This is useful for cron jobs, CI, and scripted backups.
  - It should fail fast instead of prompting for cookies.

### Scraper Quality
- Replace regex-only HTML parsing with a DOM parser such as `BeautifulSoup` or `selectolax`.
  - Current extraction is compact but brittle against small markup changes.
  - A DOM-based parser would make activity detection, image extraction, and pagination logic easier to maintain.
- Add fixture-based parser tests.
  - Save a few representative HTML pages under `tests/fixtures/`.
  - Validate parsing of original posts, activity posts, images, empty pages, and year-boundary dates.

### Efficiency
- Add optional bounded concurrency for image downloads.
  - Current image download is strictly serial with a fixed `0.5s` sleep after every image.
  - An opt-in `--image-workers 2` or `--image-workers 4` with a semaphore and shared rate limiter would improve large exports without making the default behavior more aggressive.
- Reuse one image-download implementation.
  - `douban_export.py` and `export_statuses_web.py` currently maintain separate image download logic.
  - Consolidating retry, extension detection, and failure recording reduces drift and future bugs.
